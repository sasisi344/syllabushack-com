# Hugo か Astro（それ以外）か？

## 結論：Astro 一択
**「Syllabus Hack」の構想（ブログ ＋ React製CBT試験アプリ）において、HugoよりもAstroの方が圧倒的に適しています。**

## 1. 構成の比較：なぜHugoだと辛いのか
現在検討されている「Hugo（ブログ）とReact（アプリ）を分離して管理する」という構成は、Hugoがアプリケーション開発に向いていない弱点を補うための「苦肉の策」になりがちです。

| 特徴 | Hugo (現在の構成案) | Astro (推奨) |
| :--- | :--- | :--- |
| **プロジェクト構成** | **分離管理（分断）**<br>ブログ用とアプリ用で2つのリポジトリ（またはフォルダ）を管理。<br>ビルド・デプロイ・ドメイン設定が別々になり複雑化する。 | **一元管理（統合）**<br>1つのプロジェクト内に「ブログ記事」と「Reactアプリ」が同居。<br>コマンド一発(`npm run build`)で全体がビルドされる。 |
| **アプリの埋め込み** | **サブディレクトリ運用**<br>アプリは独立したSPAとして `/app/` 等に配置。<br>ブログ記事の中に「ミニテスト」を埋め込むのは非常に困難（iframe等が必要）。 | **コンポーネント統合**<br>Reactで作ったCBTアプリを、記事(`.mdx`)の中に `<ExamApp />` と書くだけで埋め込める。<br>もちろん独立したフルスクリーンページとしても作成可能。 |
| **デザイン統一** | **難しい（二重管理）**<br>ブログ(Hugo)とアプリ(React)でCSSの管理が別々。<br>ヘッダーやフッターを両方で同じように作り直す必要がある。 | **容易（共通化）**<br>共通のCSS/Tailwind設定や、ヘッダーコンポーネントをブログとアプリの両方で使い回せる。<br>デザイン崩れの心配がない。 |
| **開発体験** | **手間が多い**<br>HugoサーバーとViteサーバーを両方立ち上げる必要がある。<br>行ったり来たりが面倒。 | **快適**<br>`npm run dev` だけでブログもReactアプリも同時にプレビューできる。<br>ホットリロードも効く。 |
| **拡張性** | **ブログ特化**<br>ブログとしては最強だが、アプリ機能の拡張には限界がある。 | **ハイブリッド**<br>ブログとアプリの境界がない。<br>「記事の途中にクイズを置く」「結果をグラフで表示する」などが自由自在。 |

## 2. Syllabus Hack での具体的メリット

### ① アプリ開発のハードルが激減する
*   **Hugo:** 「Reactアプリをビルドして、その成果物をHugoのstaticフォルダにコピーして…」というビルドパイプラインの整備が必要。
*   **Astro:** `src/components/` にReactファイルを作るだけ。即座に使えます。

### ② シームレスなユーザー体験 (UX)
*   **Hugo:** ブログ(Hugo)からアプリ(React)に移動する際、画面が完全に切り替わる（別サイト感が出る）。ヘッダーのメニューなどが微妙にズレたりしがち。
*   **Astro:** 同じ `Layout` コンポーネントを使えるため、ブログ記事からアプリページへ移動しても、全く同じサイト内を移動している感覚を維持できる。

### ③ データ連携（JSON問題データ）
*   **Hugo:** HugoのDataフォルダと、Reactアプリのsrcフォルダで、同じJSONデータをどう共有するか悩むことになる（同期スクリプトが必要など）。
*   **Astro:** プロジェクト内の `src/data/questions.json` を、ブログ記事の解説でも、CBTアプリの出題でも、直接 `import` して使える。**「ソース・オブ・トゥルース（信頼できる唯一の情報源）」を一箇所に保てる。**

## 3. 移行コストとタイミング

**今がベストタイミングです。**
Reactアプリ本体（CBTシミュレーター）の開発にまだ本格着手していないため、移行の障壁は「既存記事と画像の移動」だけです。

*   **Markdown記事:** ほぼそのまま使えます（AstroもMarkdown/MDXを標準サポート）。
*   **画像:** フォルダ移動だけで済みます。
*   **CSS:** HugoのCSSを流用するか、これを機にTailwind CSS等のモダンなスタイリングに統一することで、管理が劇的に楽になります。

## 4. 総評
「ブログを書く」だけならHugoで十分ですが、**「Reactで機能（アプリ）を作る」という要件が入った時点で、Astro（またはNext.js）を選ぶのが現代のWeb開発のスタンダード**です。

特にAstroは「コンテンツ重視のサイト（ブログ）」に向けて作られているため、Next.jsよりも学習コストが低く、Hugoからの移行先として最適です。
